## 1. 文档目的

本文档用于防止团队在执行过程中"过于发散、跑偏、做着做着变项目方向"。

目标不是限制创造力，而是保证：

- 先把核心链路做成
- 先交可运行成果
- 不把 MVP 做成无限扩张工程

---

## 2. 项目治理总则

### 总则 1：结果先于想法
团队优先交付可运行产物，不优先交付"感觉更先进"的设计。

### 总则 2：主链路优先
任何新增想法，若不能直接提高以下主链路成功率，则默认延期：

**输入图片 → OCR → 翻译 → 擦字/修复 → 回填 → 输出结果**

### 总则 3：先稳定再扩展
在主链路未稳定前，禁止把资源投入到：
- UI 美化
- 新模型探索
- 多端同时开发
- 批量调度系统
- 云服务方案

### 总则 4：引擎先于壳
先做 engine，再做插件壳、桌面壳、批处理壳。

### 总则 5：默认本地
除非明确审批，所有默认方案必须本地可运行、可控、可追踪。

### 总则 6：用真实场景驱动架构演进
架构改进必须由已发生的真实问题或已验证的场景需求触发，不允许用"未来可能会需要"作为在 MVP 阶段引入复杂架构的理由。合理的做法是：记录储备方案（ADR），明确触发条件，等条件成立再启动。

---

## 3. 项目北极星

本项目当前唯一北极星不是"做一个很厉害的 AI 产品"，而是：

**把已有旧脚本能力升级为稳定、可复用、可被插件调用的本地图片翻译引擎。**

任何偏离这一句的话题，都必须接受范围审查。

---

## 4. 范围治理机制

## 4.1 三层需求分类

### A 类：必须做
满足以下任一条件即归类 A：
- 直接提升主链路成功率
- 直接消除阻塞
- 是本轮验收必需项
- 是后续插件调用的必要前置

示例：
- 配置中心
- OCR provider 接口
- `/process` API
- OpenCV 回退机制
- ComfyUI 熔断降级（低成本鲁棒性提升）
- 请求超时保护

### B 类：可延后做
- 对体验有帮助，但不影响 MVP 跑通
- 属于增强项，不属于阻塞项

示例：
- 更丰富的字体策略
- 批量多图处理
- 浏览器 popup 美化

### C 类：禁止本轮做
- 改变产品形态
- 会显著扩张范围
- 不是当前里程碑验收项

示例：
- SaaS 化
- 用户系统
- 云端翻译平台
- 视频翻译
- 多人协作后台
- 商业化支付
- **异步任务队列系统（已记录为 ADR-006 储备方案）**
- **VRAM 显存互斥管理器（已记录为 ADR-006 储备方案）**

---

## 5. 员工防跑偏规则

## Rule-01：任何新增需求必须先归类
不能一句"我顺手一起做了"就进入开发。
新增事项必须标记：
- A / B / C 分类
- 是否影响当前里程碑
- 是否需要审批

## Rule-02：禁止把"技术兴趣"伪装成"项目需要"
常见伪装说法：
- "我顺便把架构升级一下"
- "我顺便支持多模型"
- "我顺便把插件也做好"
- "我顺便做个队列系统"
- "我顺便加个 web 页面"
- "我顺便把 API 改成异步的"
- "我顺便加个显存管理器"

凡是"顺便"二字，默认高风险。

## Rule-03：不允许边做边改项目目标
项目目标只能在里程碑复盘会上调整，不能在开发中由单个员工私自修改。

## Rule-04：任何跨层开发必须说明理由
例如：
- 算法同学要改插件前端
- 前端同学要设计 engine 架构
- QA 同学要新增模型 provider

必须先说明：
1. 为什么当前 owner 不能完成
2. 为什么现在必须跨层
3. 是否会影响里程碑

## Rule-05：没有验收证据不算完成
任何任务如果没有最小证据，视为未完成。
证据包括：
- 运行日志
- 输出图
- JSON 结果
- 测试记录
- README 说明

## Rule-06：技术建议必须走审批流程，不能绕过治理体系直接落地
即使技术建议本身合理，也必须经过需求分类（A/B/C）和决策门禁审查。正确流程是：提出建议 → 归类 → 审批 → 排期。不允许以"技术上更先进"为由跳过治理流程。

---

## 6. 决策门禁（Approval Gates）

以下事项未经审批不得进入开发：

1. 新增第三方模型
2. 新增云端依赖
3. 新增浏览器正式功能
4. 新增 GUI 大前端
5. 新增异步任务系统
6. 新增数据库
7. 改变 API 协议（含同步改异步）
8. 新增项目子方向
9. **启动已记录的储备方案（如 ADR-006 异步队列/VRAM 互斥），须验证触发条件已成立**

---

## 7. 里程碑门禁

## Gate-M1：Engine Skeleton Gate
通过条件：
- 模块边界清晰
- 去除硬编码
- 统一配置加载
- 有单入口 process()

不通过情形：
- 仍以单脚本堆逻辑
- 仍依赖固定图片路径
- 模块输入输出不统一

## Gate-M2：Local API Gate
通过条件：
- `/process` 可跑通
- 返回结构化 JSON
- Fast 模式稳定
- 请求超时保护生效
- 单请求互斥锁生效
- ComfyUI 熔断降级可触发

不通过情形：
- 只能手工改脚本运行
- 结果不可复现
- 失败不报错、不回退
- 无超时保护导致请求挂死

## Gate-M3：HQ Mode Gate
通过条件：
- ComfyUI 模式可调用
- 失败时自动回退
- 熔断降级经过测试
- 有质量报告

不通过情形：
- 只在作者机器偶然能跑
- 没有调试产物
- 没有 fallback
- 熔断降级未验证

## Gate-M4：Browser Contract Gate
通过条件：
- 明确插件到本地服务的接口协议
- 定义最小交互流
- 完成是否需要异步改造的评估

不通过情形：
- 先做大量 UI
- 先做商店上架
- 没有协议就直接写插件
- 未评估异步需求就盲目实现

---

## 8. 周节奏管理

建议固定节奏：

### 周初
- 锁定本周目标
- 冻结范围
- 明确 owner

### 周中
- 只汇报阻塞、风险、偏差
- 不开脑洞会

### 周末
- 只看是否达成 DoD
- 未完成项回炉，不以"差不多"通过

---

## 9. 会议规则

### 允许讨论的话题
- 主链路阻塞
- 模块接口
- 测试失败
- 风险与取舍

### 禁止在例会中发散的话题
- 商业化路线
- 多产品延伸
- UI 风格争论
- 平台大战
- "未来也许会需要"的功能

### 一句判断法
如果一个议题不能回答：
**"它怎么提高本里程碑通过率？"**
那就先不讨论。

## 10. 任务卡模板

每个任务必须用以下模板：

### 任务名称
一句话写清楚结果，不写愿景。

### 目标
要解决什么问题。

### 输入
依赖什么。

### 输出
交付什么文件、代码、接口、结果。

### 完成标准
怎样算 done。

### 不做什么
明确边界。

### 风险
可能失败点。

### Owner
唯一负责人。

## 11. PM 执行准则

PM 的职责不是"鼓励大家自由发挥"，而是：

1. 看守边界
2. 拦截发散
3. 拒绝伪需求
4. 把大目标拆成阶段目标
5. 只用验收标准讲话

PM 在本项目中必须高频使用三句话：

- "这件事是否属于当前里程碑？"
- "这件事是否提升主链路成功率？"
- "这件事没有它，当前版本是否无法验收？"

只要三问里有两问回答是否定，就延期。

---

## 12. 架构师执行准则

架构师不能陷入三种典型跑偏：

### 跑偏 1：过度抽象
症状：
- provider 还没落地，就抽象出 10 层接口
- 工程复杂度远超 MVP 需求
- 为尚不存在的并发场景设计资源调度器

对策：
- 本轮只允许为已有能力抽象，不允许为假想未来过度抽象
- 超出当前需求的架构方案，以 ADR 形式记录储备，明确触发条件，不实现
- 架构建议必须回答："当前并发数为 1 的前提下，这个组件会被触发吗？"如果答案是否定的，则记录但不实现

### 跑偏 2：技术洁癖
症状：
- 追求绝对完美目录
- 拒绝阶段性妥协
- 无限重构

对策：
- 优先"可跑 + 可测 + 可接入"
- 不以"还不够优雅"为延迟理由

### 跑偏 3：绕过治理体系推进技术方案
症状：
- 直接给出"替换掉第 X 节"的完整方案，绕过需求分类和审批流程
- 以"技术上更合理"为由，要求在 MVP 阶段引入复杂架构组件
- 把架构评审意见等同于已审批的执行指令

对策：
- 技术建议必须走 Rule-06 流程：提出 → 归类 → 审批 → 排期
- 有价值的技术判断以 ADR 形式收编，不直接进入开发
- PM 有权将架构建议中的合理部分拆解为低成本任务（如熔断降级），同时将高成本部分（如异步队列）推迟到有真实场景驱动时再评估

---

## 13. QA 执行准则

QA 不是最后找 bug 的人，而是防跑偏的最后一道门。

QA 必查项：
1. 是否越界开发
2. 是否引入未审批依赖
3. 是否保留硬编码
4. 是否缺少回退
5. 是否缺测试证据
6. 是否在 MVP 阶段引入了超出范围的架构组件（如异步队列、显存管理器）

如发现越界，可直接打回，不需要等 PM 同意。

---

## 14. 偏差处理机制

发现跑偏后，按以下流程执行：

### Level 1：轻微偏差
例：多写了一个无伤大雅的辅助工具
处理：记录，不扩散，要求补齐主任务

### Level 2：中度偏差
例：开始做不在本轮范围的插件 UI
处理：停止分支开发，回到主线

### Level 3：重大偏差
例：私自改项目方向，新增云服务/数据库/复杂系统
处理：
- 停止合并
- 单独评审
- 必要时废弃该分支成果

---

## 15. 变更控制

只有以下场景允许调整里程碑：

1. 当前路径被验证不可行
2. 外部约束发生变化
3. 现有验收标准明显不合理
4. 有明确数据证明改动收益显著

禁止因为：
- 某员工更想做别的
- 某技术看起来更新
- 某功能"感觉以后会用到"
- 某架构方案"技术上更先进"（但 MVP 阶段无真实需求驱动）

---

## 16. 红线清单

以下行为直接视为违规：

1. 未审批新增云依赖
2. 未审批改变范围
3. 未审批引入数据库/队列系统
4. 未完成主线却优先做 UI
5. 无测试证据就宣称完成
6. 把个人试验代码合入主线
7. 用"未来扩展性"作为无限抽象借口
8. **绕过治理流程直接将架构建议作为执行指令落地**

---

## 17. 本项目最重要的一句话

**宁可先做出一个可被插件调用的稳定本地引擎，也不要做出一个看起来很大、实际上主链路不稳的空架子。**

## 附录 A：范围审查案例

### 案例 1：异步任务队列与 VRAM 互斥锁提案

**背景**：团队内部架构评审中，有成员提出将 API 从同步模式改为全异步提交 + 状态轮询模式，并引入 VRAM 显存互斥锁和 Watchdog 僵尸任务清理机制。

**技术判断**：该提案中的技术观点是合理的——在多模型并发场景（如 Ollama LLM + ComfyUI 同时抢占 GPU）下，显存互斥和异步队列确实是必要的架构组件。ComfyUI 熔断降级的建议也具备实际价值。

**范围审查结论**：

对照治理规则逐项审查：

| 提案内容 | 触发的门禁/规则 | 判定 |
|----------|-----------------|------|
| API 同步改异步 | 决策门禁第 7 条：改变 API 协议 | C 类，本轮不做 |
| 异步任务队列 | 决策门禁第 5 条：新增异步任务系统 | C 类，本轮不做 |
| VRAM 显存互斥锁 | Rule-02：MVP 并发数=1，该组件不会被触发 | C 类，本轮不做 |
| Watchdog 僵尸清理 | 依赖异步队列，属于连带范围膨胀 | C 类，本轮不做 |
| ComfyUI 熔断降级 | 直接提升主链路鲁棒性，低实现成本 | A 类，本轮实现 |
| 请求超时保护 | 直接消除阻塞风险，低实现成本 | A 类，本轮实现 |

**处理方式**：
1. 将异步队列和 VRAM 互斥作为 ADR-006 储备方案记录，明确触发条件
2. 将 ComfyUI 熔断降级作为 ADR-007 纳入本轮实现（Task List T3.5）
3. 将请求超时保护纳入本轮实现（Task List T3.4）

**本案例的治理意义**：这是治理体系的第一个实际测试用例。它验证了一个关键原则——承认技术判断有价值，不等于接受全部方案进入当前迭代。正确做法是用治理流程拆解提案，收编合理部分，推迟超出范围的部分，并为后者设定明确的启动条件。
